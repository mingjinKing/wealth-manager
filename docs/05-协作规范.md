## 一、gstack 命令总览

## 二、标准开发流程

                  ↓

│     合并 → 版本升级 → CHANGELOG → PR           │

## 三、Claude Code + gstack 实战工作流

### 3.1 日常开发会话

```
用户（通过乘黄 AI 助手）
    ↓ 指令（cc_send/cc_plan/cc_execute）
Claude Code（gstack 环境）
    ↓
执行任务（如 /qa、/browse、代码编辑等）
    ↓
结果返回 GitHub（代码变更、文档更新）
```

### 3.2 Claude Code 具体使用场景

#### 场景1：文档全面更新

```
用户指令："为知财项目全面更新 GitHub 文档"
    ↓
Claude Code 执行：
  1. 读取相关代码文件（Color.kt、Typography.kt、DashboardScreen.kt 等）
  2. 读取现有文档
  3. 对齐代码与文档
  4. 按优先级更新各文档
  5. git add → git commit → git push
```

#### 场景2：功能开发

```
用户指令："实现预算管理功能"
    ↓
Claude Code 执行：
  1. /autoplan 三方评审
  2. 评审结果记录到 docs/REVIEW/
  3. 编码实现
  4. /qa 验证
  5. /ship 发布
```

#### 场景3：Bug 修复

```
用户指令："首页 FAB 位置偏移"
    ↓
Claude Code 执行：
  1. /browse 打开 App 复现问题
  2. /investigate 根因分析
  3. 定位代码问题
  4. 修复并验证
```

### 3.3 评审结果记录规范

**重要**：所有评审结果必须记录到 `docs/REVIEW/` 目录

```
docs/
├── REVIEW/
│   ├── 2026-04-02-预算管理-autoplan.md
│   ├── 2026-04-02-预算管理-ceo-review.md
│   ├── 2026-04-02-预算管理-eng-review.md
│   └── 2026-04-02-预算管理-design-review.md
```

**评审报告命名规范**：
```
{日期}-{功能名}-{评审类型}.md
```

**评审报告内容规范**：
```markdown
# {功能名} - {评审类型}

## 评审日期
{YYYY-MM-DD}

## 评审结论
{PASS / 需要修改 / 阻塞}

## 主要意见
1. ...

## 决策记录
- 决策1：...
- 决策2：...

## 待解决问题
- 问题1：...
```

## 四、产品设计分歧上报流程

### 4.1 分歧定义

产品设计分歧包括但不限于：
- UI/UX 方案选择争议
- 功能优先级排序争议
- 技术实现与产品预期不一致
- 用户价值判断分歧

### 4.2 处理流程

```
┌─────────────────────────────────────────────┐
│  发现分歧                                   │
│  （Claude Code 或团队成员）                  │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  记录分歧到 docs/REVIEW/                     │
│  格式：{日期}-{模块}-分歧.md                  │
│  包含：各方观点、论据、待决策问题              │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│ 上报用户决策                                 │
│  （通过乘黄 AI 助手 / 飞书群）                 │
│  选择题形式呈现，不超过3个选项                 │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  用户拍板                                   │
│  决策记录到 docs/REVIEW/                    │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  继续执行                                   │
└─────────────────────────────────────────────┘
```

### 4.3 分歧报告模板

```markdown
# {功能名} - 设计分歧

## 分歧日期
{YYYY-MM-DD}

## 分歧类型
- [ ] UI/UX 方案
- [ ] 功能优先级
- [ ] 技术与产品预期
- [ ] 用户价值判断
- [ ] 其他

## 各方观点

### 观点A：{提出方}
**立场**：...
**论据**：
1. ...
2. ...

### 观点B：{提出方}
**立场**：...
**论据**：
1. ...
2. ...

## 待决策问题
1. ...

## 上报用户选项
1. 选项1：...
2. 选项2：...
3. 选项3：...

## 用户决策
（等待用户填写）
```

## 五、文档与代码的关系

| 信息类型 | 存储位置 | 说明 |
|----------|----------|------|
| 产品需求 | `docs/` | 唯一产品知识来源，GitHub 托管 |
| 技术规格 | `SPEC.md` | 技术实现规格 |
| 评审记录 | `docs/REVIEW/` | 所有评审结果归档 |
| 代码 | `/` 根目录 | 各模块代码 |

**重要原则**：产品文档和 GitHub 代码是唯一的两个目标产品知识来源。所有需求必须来自 `docs/` 目录，而非记忆或外部工具。
