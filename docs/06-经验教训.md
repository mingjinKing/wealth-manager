## 一、工作流沉淀

### 1.1 文档与代码统一管理

| 信息类型 | 存储位置 | 说明 |
|----------|----------|------|
| 产品需求文档 | `docs/` | 唯一产品知识来源，GitHub 托管 |

**原则**：产品文档和 GitHub 代码是唯一的两个目标产品知识来源。

```

openclaw-cc-bridge（cc_send / cc_plan / cc_execute）
    ↓
```

**工具链**：
- OpenClaw：AI 助手中枢

- Claude Code：编码引擎
- gstack：虚拟团队 skill 体系（autoplan、qa、ship 等）

- 评审记录统一存放在 `docs/REVIEW/` 目录

## 二、踩过的坑

### 2.1 键盘覆盖问题（adjustResize + imePadding + weight + align）

**问题描述**：
在 Android Compose 中使用数字键盘输入时，键盘弹出可能会遮挡底部输入区域。

**根本原因**：
- `adjustResize` 模式下，窗口会调整大小来为键盘腾出空间
- 但如果布局使用了 `weight` 来分配高度，且父布局没有正确处理 `imePadding()`
- 或者是 `align` 对齐问题导致元素位置计算错误

**正确解决方案**：

```kotlin
// 1. 确保 Scaffold 的 contentPadding 或 containerColor 处理了键盘
Scaffold(
    containerColor = MaterialTheme.colorScheme.background
) { paddingValues ->
    Column(
        modifier = Modifier
            .fillMaxSize()
            .padding(paddingValues)
            .navigationBarsPadding()
            .imePadding()  // 关键：添加 imePadding
    ) {
        // 内容区域
    }
}

// 2. 底部固定区域使用 weight 时，确保父布局正确计算
Row(
    modifier = Modifier
        .fillMaxWidth()
        .imePadding()  // 添加在父级
) {
    // weight 元素会正确响应键盘弹出
}

// 3. 避免在 weight 区域内使用 align 问题
// 错误：align 在 weight 区域内可能导致计算错误
// 正确：align 应该在 weight 计算之前完成
```

**经验总结**：
- `imePadding()` 必须添加到键盘弹出时需要调整的区域
- `weight` 和 `align` 不要混用，先 align 再 weight
- 测试时使用真实键盘（输入法）而非硬键盘

### 2.2 多个 git clone 目录导致混淆

**问题描述**：
同一个项目在本地有多个 clone 目录（如 `wealth-manager/`、`wealth-manager-copy/`、`wealth-manager-new/`），导致：
- 不知道哪个是最新版本
- 在错误的目录修改代码
- git 操作指向错误的工作区
- 文档更新后发现没更新到正确的仓库

**根本原因**：
- 没有统一的本地开发目录规范
- 随意创建副本进行"备份"或"实验"
- 没有清理过期的工作区

**正确解决方案**：

```bash
# 查看所有 wealth-manager 相关目录
find ~ -type d -name "wealth-manager*" 2>/dev/null

# 删除不需要的副本
rm -rf ~/path/to/old-copy

# 保持只有一个工作目录
~/repo/wealth-manager/
```

**预防措施**：
1. **一个项目只维护一个工作目录**
2. **使用 git branch 进行实验**，而非复制目录
3. **使用 git worktree 进行并行工作**，而非复制目录
4. **定期清理**：`git remote -v` 确认远程仓库正确

**经验总结**：
- 副本目录不会帮你"保险"，反而会增加混乱
- git 是你最好的版本控制工具，善用 branch 和 worktree
- 开始工作前先 `pwd` 和 `git status` 确认位置

### 2.3 文档发散

**问题**：产品文档初期包含过多扩展方向（社区、游戏化、多平台等），导致定位不清。

**解决**：采用 V1 聚焦策略，删除非 V1 功能，聚焦"记账 + 透视"核心。

**原则**：先做减法，再做乘法。

### 2.4 文档与代码不同步

**问题**：文档存在但未与代码仓库同步，导致信息孤岛。

**解决**：所有产品文档统一托管于 GitHub，与代码在同一个仓库。

**原则**：单一信息源，避免重复维护。

### 2.5 评审流程缺失

**问题**：跳过评审直接开发，导致返工。

**解决**：使用 `/autoplan` 进行 CEO + 工程 + 设计三方评审后再开始编码。

**原则**：磨刀不误砍柴工，评审的边际成本远低于返工。

### 2.6 混淆 Wiki 与代码库文档的职责边界

**问题描述**：
收到「更新规则引擎现状」的任务后，错误地认为要更新飞书 Wiki，实际上：
- Wiki 职责：只记录**产品哲学、理念、未实现的规划 idea**（以长官思想为主）
- 代码库文档职责：**实现细节、现状梳理、硬编码分布**（在哪里实现就在哪里记录）

**根本原因**：
- 没有严格遵循既定分工原则，看到「文档」就默认想到 Wiki
- 没有先确认职责边界就动手
- 对「现状」和「规划」的归属判断错误：
  - 「现状」（当前代码里有什么）→ 代码库 docs/
  - 「目标架构」（未来要做什么）→ Wiki 第十章（产品哲学延伸）

**正确理解**：
| 内容 | 存储位置 | 负责方 |
|------|----------|--------|
| 产品哲学、理念 | 飞书 Wiki | 长官 |
| 未实现的规划 idea | 飞书 Wiki | 长官 |
| 实现细节、硬编码分布 | 代码库 docs/ | 乘黄整理 / Claude Code 更新 |
| 目标架构（旺财引擎愿景） | 飞书 Wiki 第十章 | 长官主导 |

**教训**：
- 收到文档类任务时，先问：**这是「Philosophy/Idea」（Wiki）还是「Implementation/Status」（代码库）？**
- 规划类产品目标准确记 Wiki，实现类现状老老实实写代码库文档
- 乘黄的判断标准不是「任务内容」，而是**信息的最终归属**

### 2.7 规则引擎统一口径

**日期**：2026-04-02

**背景**：知财 v1.2 开始构建"旺财"引擎，需先统一对"规则"的定义。

**规则定义**：
> 规则 = 有明确**触发条件** + 产生一个**结论/动作**的逻辑。

按此口径，当前 8 条硬编码规则重新分类：

| # | 内容 | 是否规则 | 原因 |
|---|------|----------|------|
| 1 | 哇时刻触发 `savedAmount > 100 && savedAmount > avg × 20%` | ✅ 是 | 条件判断，触发成就卡 |
| 2 | 规模分析 `total > 5000` | ⚠️ 是规则，但阈值待改 | 条件判断，但 5000 是拍脑袋，应与预算关联 |
| 3 | 结构偏向 `top.percentage > 0.4` | ✅ 是 | 条件判断，触发结构建议 |
| 4 | 频率分析 `totalCount > 50` | ✅ 是 | 条件判断，触发频率建议 |
| 5 | 预算类型判断（if 月预算 > 0 else 周预算） | ❌ 不是 | 纯数据获取，无触发条件 |
| 6 | 净资产计算 `assets + liabilities` | ❌ 不是 | 纯数学计算，无判断 |
| 7 | 收支区分 `category.type == "INCOME"` | ❌ 不是 | 数据分类，无判断 |
| 8 | trend 等分 5 段 | ❌ 不是 | 固定常量，纯粹数学 |

**结论**：真正算规则的只有 1、3、4；规则 2 待改（阈值与预算挂钩）。

**非规则的计算逻辑**不放入 rules/ 包，作为 Calculator 或工具函数维护。

## 三、方法论沉淀

### 3.1 gstack 虚拟团队工作流

```
┌─────────────────────────────────────────────┐
│  1. 需求确认                                  │
│     阅读 docs/ 目录下的产品文档                  │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  2. /autoplan 三方评审                        │
│     CEO + 工程 + 设计 联合评审                  │
│     输出：评审报告 → docs/REVIEW/              │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  3. 编码实现                                  │
│     基于评审结果开发 SPEC.md 功能                │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  4. /qa 验证                                 │
│     功能测试 + 回归测试                         │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  5. /ship 发布                               │
│     合并 → 版本升级 → CHANGELOG → PR           │
└─────────────────────────────────────────────┘
```

### 3.2 渐进式披露原则

产品开发采用渐进式披露：
1. **V1**：核心功能验证（记账 + 透视 + 资产管理 + 预算管理）
2. **V2**：平台导入（数据打通）+ 周统计
3. **V3**：社区功能（用户互动）

每个版本只披露当前阶段相关信息，避免信息过载。

### 3.3 产品设计分歧处理

当出现产品设计分歧时：
1. 记录分歧到 `docs/REVIEW/{日期}-{模块}-分歧.md`
2. 上报用户决策（选择题形式，不超过3个选项）
3. 用户拍板后继续执行
