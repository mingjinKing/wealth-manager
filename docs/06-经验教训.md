## 一、工作流沉淀

### 1.1 文档与代码统一管理

| 信息类型 | 存储位置 | 说明 |
|----------|----------|------|
| 产品需求文档 | `docs/` | 唯一产品知识来源，GitHub 托管 |
| 技术规格 | `SPEC.md` | 技术实现规格 |
| 代码 | `/` 根目录 | 各模块代码 |

**原则**：产品文档和 GitHub 代码是唯一的两个目标产品知识来源。

### 1.2 OpenClaw → Claude Code → GitHub 工作流

```
飞书（乘黄 · AI 助手）
    ↓ 指令
openclaw-cc-bridge（cc_send / cc_plan / cc_execute）
    ↓
Claude Code + gstack（虚拟团队）
    ↓
GitHub（代码 + 文档）
```

**工具链**：
- OpenClaw：AI 助手中枢
- openclaw-cc-bridge：OpenClaw 与 Claude Code 的桥接插件
- Claude Code：编码引擎
- gstack：虚拟团队 skill 体系（autoplan、qa、ship 等）
- GitHub：代码和文档的版本控制

### 1.3 文档版本管理

- 所有文档变更通过 Pull Request 审核后合并
- 文档版本与代码版本保持同步
- 文档编号：01、02、03... 按生命周期排序

## 二、踩过的坑

### 2.1 文档发散

**问题**：产品文档初期包含过多扩展方向（社区、游戏化、多平台等），导致定位不清。

**解决**：采用 V1 聚焦策略，删除非 V1 功能，聚焦"记账 + 透视"核心。

**原则**：先做减法，再做乘法。

### 2.2 文档与代码不同步

**问题**：文档存在但未与代码仓库同步，导致信息孤岛。

**解决**：所有产品文档统一托管于 GitHub，与代码在同一个仓库。

**原则**：单一信息源，避免重复维护。

### 2.3 评审流程缺失

**问题**：跳过评审直接开发，导致返工。

**解决**：使用 `/autoplan` 进行 CEO + 工程 + 设计三方评审后再开始编码。

**原则**：磨刀不误砍柴工，评审的边际成本远低于返工。

## 三、方法论沉淀

### 3.1 gstack 虚拟团队工作流

```
┌─────────────────────────────────────────────┐
│  1. 需求确认                                  │
│     阅读 docs/ 目录下的产品文档                  │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  2. /autoplan 三方评审                        │
│     CEO + 工程 + 设计 联合评审                  │
│     输出：评审报告 + 决策记录                    │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  3. 编码实现                                  │
│     基于评审结果开发 SPEC.md 功能                │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  4. /qa 验证                                 │
│     功能测试 + 回归测试                         │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  5. /ship 发布                               │
│     合并 → 版本升级 → CHANGELOG → PR           │
└─────────────────────────────────────────────┘
```

### 3.2 渐进式披露原则

产品开发采用渐进式披露：
1. **V1**：核心功能验证（记账 + 透视）
2. **V2**：平台导入（数据打通）
3. **V3**：社区功能（用户互动）

每个版本只披露当前阶段相关信息，避免信息过载。
