### 1.1 Room 数据库 5张表 ✅ PASS

| 表名 | SPEC.md 定义 | 代码实现 | 一致性 |
|------|-------------|---------|--------|
| CategoryEntity | id, name, icon | id, name, icon, color, type, isDefault | ✅ 扩展字段（向后兼容）|

**详细说明：**

#### ExpenseEntity type 字段问题 ⚠️ WARN

- **SPEC.md** 描述：`type: Int`（0=支出，1=收入）
- **实际代码**：`ExpenseEntity` 没有 `type` 字段，区分收入/支出通过 `CategoryEntity.type`（"EXPENSE"/"INCOME"）实现
- **评估**：这是一个**设计层面的差异**，代码实现实际上更合理（通过分类而非每条记录存储类型），但 SPEC 文档未更新

#### WeekStatsEntity 字段名差异 ⚠️ WARN

- **SPEC.md**：使用 `totalSpend`（周消费总额）
- **实际代码**：使用 `totalAmount`（周消费总额）
- **评估**：语义相同但命名不一致，需统一

### 3.2 哇时刻公式 ❌ BLOCK

**SPEC.md 公式**（07-UI设计规格.md 第十一节）：
```
savedAmount = last_4_week_avg - current_week_spend
wow_triggered = savedAmount > ¥100 AND savedAmount > last_4_week_avg × 20%
```

**实际代码**（DashboardViewModel.kt 第96-109行）：
```kotlin
val avgLast4Weeks = recentStats.take(4).map { it.totalAmount }.average()
    .takeIf { !it.isNaN() } ?: recent7DaysTotalExpense
val savedAmount = avgLast4Weeks - recent7DaysTotalExpense

// 降低触发门槛以便展示激励层，或根据用户需求"启用"
val isTriggered = savedAmount > 0  // ❌ 简化版本！
val wowPreview = if (isTriggered) {
    WowPreview(
        savedAmount = savedAmount,
        isTriggered = true,
        reason = if (savedAmount > 100) "本周消费控制极佳！" else "继续保持良好的消费习惯"
    )
} else null
```

**问题分析：**
1. SPEC 要求两个条件同时满足（`savedAmount > ¥100` AND `savedAmount > last_4_week_avg × 20%`）
2. 代码仅检查 `savedAmount > 0`
3. 这会导致哇时刻**更容易触发**，降低了激励门槛

**影响评估：**
- 严重程度：**中等**（用户体验影响，非功能错误）
- 建议：按 SPEC 恢复完整公式，或确认简化是产品决策
