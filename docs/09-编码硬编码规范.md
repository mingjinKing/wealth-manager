## 二、问题分析

### 2.1 严重程度

| 等级 | 问题 | 数量 |
|------|------|------|
| 🔴 高 | API 地址、密钥等敏感配置硬编码 | 5 |
| 🟡 中 | 业务阈值硬编码，无法动态调整 | 5 |
| 🟢 低 | UI 常量、显示阈值 | 4 |

### 2.2 风险点

1. **API 地址硬编码**
   - 如果服务器 IP 变化，需要修改代码
   - 建议：放入配置文件或 BuildConfig

2. **业务阈值硬编码**
   - 运营过程中需要调整阈值（如哇时刻比例）
   - 需要重新编译 App
   - 建议：放入 SharedPreferences 或远程配置

3. **超时时间分散**
   - `LLMModule.kt` 和 `LLMClient.kt` 有不同的超时配置
   - 建议：统一管理

## 四、规范要求

### 4.1 新增常量

新增任何常量时，必须满足：

1. **命名规范**：`UPPER_SNAKE_CASE`
2. **文档注释**：说明用途和默认值
3. **分类管理**：按功能模块分组

```kotlin
/**
 * 哇时刻规则配置
 */
object WowRuleConfig {
    /** 最低节省金额阈值（元）*/
    const val MIN_SAVED_AMOUNT = 100.0
    
    /** 相对节省比例阈值（相对于历史均值）*/
    const val SAVED_RATIO_THRESHOLD = 0.2
}
```

### 4.2 禁止事项

| 禁止 | 替代方案 |
|------|----------|
| 在业务代码中直接写数字 | 使用命名常量 |
| 多处重复相同的超时值 | 提取到统一配置类 |
| 敏感信息硬编码 | 使用 BuildConfig 或加密存储 |
| API 地址分散在多处 | 统一在配置类中管理 |

### 4.3 现有硬编码处理

| 状态 | 处理方式 |
|------|----------|
| API 地址 | ✅ 已通过 BuildConfig 管理 API Key，超址待迁移 |
| 业务阈值 | 🟡 保持现状，后续迭代中逐步迁移到配置中心 |
| UI 常量 | 🟢 保持现状，已通过 DesignTokens 统一管理部分 |
