## 一、gstack 命令总览

### 核心开发流程

| 命令 | 说明 | 使用时机 |
|------|------|----------|
| `/autoplan` | 自动评审管道（CEO + 设计 + 工程三方评审）| 开始编码前必跑 |
| `/plan-ceo-review` | CEO/Founder 评审 | 战略、scope、野心 |
| `/plan-eng-review` | 工程负责人评审 | 架构、技术选型、测试覆盖 |
| `/plan-design-review` | 设计师评审 | UI/UX、设计维度 |
| `/codex review` | OpenAI Codex 独立审查 | 第二意见 |

### 编码与测试

| 命令 | 说明 | 使用时机 |
|------|------|----------|
| `/browse` | 头戴浏览器 QA 测试 | 验证 UI、功能 |

| `/guard` | 安全模式（危险操作警告）| 高风险操作前 |

### 代码质量

| 命令 | 说明 | 使用时机 |
|------|------|----------|

| `/simplify` | 代码简化与优化 | 重构时 |
| `/investigate` | 根因调试 | Bug 排查 |
| `/cso` | 安全审计 | 上线前 |

### 发布与部署

| 命令 | 说明 | 使用时机 |

| `/ship` | 发布工作流（合并→测试→升级→PR）| 准备发布 |

| `/canary` | 金丝雀监控 | 发布后 |

## 二、标准开发流程

```
┌─────────────────────────────────────────────┐
│  1. 需求确认                                  │
│     阅读 docs/ 目录下的产品文档                  │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  2. /autoplan 三方评审                        │
│     CEO + 工程 + 设计 联合评审                  │
│     输出：评审报告 + 决策记录                    │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  3. 编码实现                                  │
│     基于评审结果开发 SPEC.md 功能                │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  4. /qa 验证                                 │
│     功能测试 + 回归测试                         │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  5. /cso 安全审计                             │
│     安全检查 + 合规验证                         │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  6. /ship 发布                               │
│     合并 → 版本升级 → CHANGELOG → PR           │
└─────────────────────────────────────────────┘
```

## 三、文档与代码的关系

| 信息类型 | 存储位置 | 说明 |
|----------|----------|------|
| 产品需求 | `docs/` | 唯一产品知识来源，GitHub 托管 |
| 技术规格 | `SPEC.md` | 技术实现规格 |
| 代码 | `/` 根目录 | 各模块代码 |

**重要原则**：产品文档和 GitHub 代码是唯一的两个目标产品知识来源。所有需求必须来自 `docs/` 目录，而非记忆或外部工具。
